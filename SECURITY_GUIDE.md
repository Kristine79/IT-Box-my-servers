## Что защищает ваши данные

### 1. Военное шифрование данных 🔐

**AES-256-GCM** — это тот же алгоритм шифрования, который использует:
- Правительство США для секретной информации
- Банки для защиты транзакций
- Apple для iCloud
- Google для данных пользователей

**Как это работает:**
- Все пароли и чувствительные данные шифруются перед сохранением
- Каждое шифрование использует уникальный ключ (96-битный IV)
- 256-битный ключ невозможно взломать перебором даже с суперкомпьютерами
- Даже если база данных будет скомпрометирована — данные останутся зашифрованными

### 2. Защита от взлома и атак 🛡️

| Угроза | Защита |
|--------|--------|
| **Перебор паролей** | Ограничение: 5 попыток за 15 минут с одного IP |
| **Внедрение вредоносного кода** | Автоматическая проверка всех входных данных |
| **Перехват данных** | HTTPS/TLS 1.3 для всех соединений |
| **Атаки на API** | Rate limiting: максимум 100 запросов в минуту |
| **Подделка запросов** | Проверка всех заголовков и токенов |

### 3. Безопасная аутентификация 👤

- **Firebase Authentication** — инфраструктура от Google
- **JWT-токены** с автоматическим истечением срока
- **Email verification** — подтверждение email перед доступом
- **Сессии с ограниченным временем** — автоматический выход

### 4. Многоуровневая защита (Defense in Depth)

```
Пользователь
    ↓
HTTPS/TLS 1.3 (шифрование канала)
    ↓
Rate Limiting (защита от DDoS)
    ↓
Authentication (проверка личности)
    ↓
Input Validation (очистка данных)
    ↓
Authorization (проверка прав доступа)
    ↓
AES-256-GCM Encryption (шифрование данных)
    ↓
Firestore Database (защищенное хранилище Google)
```

## Как защищены ваши сервера и пароли

### Шифрование на стороне сервера

```
Ваш пароль → [AES-256-GCM] → Зашифрованный текст → Firestore
                 ↑
         256-битный ключ (только сервер)
```

**Важно:** Ключ шифрования никогда не покидает сервер. Даже разработчики не могут прочитать ваши пароли.

### Что видит приложение:

| Что хранится | Как хранится |
|--------------|--------------|
| Пароли серверов | Зашифровано AES-256-GCM |
| Данные доступов | Зашифровано AES-256-GCM |
| Логины/Email | Открытый текст (нужен для входа) |
| Метаданные серверов | Открытый текст (названия, IP) |

## Защита от конкретных угроз

### SQL-инъекции и взлом через формы

- ✅ Все входные данные проверяются на опасные паттерны
- ✅ Максимальная длина ограничена (5000 символов)
- ✅ Спецсимволы автоматически экранируются
- ✅ Типы данных строго валидируются

### XSS-атаки (внедрение скриптов)

- ✅ Заголовок `X-XSS-Protection` (отключен в пользу CSP)
- ✅ Content Security Policy блокирует inline-скрипты
- ✅ Все пользовательские данные санитизируются

### CSRF-атаки (подделка запросов)

- ✅ SameSite cookies
- ✅ Проверка Origin и Referer
- ✅ CORS-политика ограничивает домены

### DDoS и перегрузка

- ✅ Rate limiting на всех API endpoints
- ✅ Ограничение размера запросов (10KB макс)
- ✅ Автоматическая блокировка подозрительных IP

## Мониторинг безопасности

### Автоматический детектинг угроз

Система отслеживает:
- 🔍 Необычное количество попыток входа
- 🔍 Доступ с новых IP-адресов в необычное время
- 🔍 Подозрительные паттерны в запросах
- 🔍 Попытки ввода вредоносного кода

**Реакция:**
- Автоматическая блокировка при >5 неудачных входов за 15 минут
- Логирование всех security-событий
- Генерация security-отчетов

## Что это значит для вас

### ✅ Ваши данные в безопасности, потому что:

1. **Даже при взломе сервера** — злоумышленник получит только зашифрованные данные
2. **Даже при перехвате трафика** — HTTPS защищает от прослушивания

## Вопросы и ответы

**Q: Могут ли разработчики увидеть мои пароли?**
A: Нет. Пароли шифруются на сервере ключом, который не доступен разработчикам.

**Q: Что если сервер взломают?**
A: Злоумышленник получит зашифрованные данные. Без ключа шифрования они бесполезны.

**Q: Насколько надежно AES-256?**
A: Это тот же уровень шифрования, что используется для государственной тайны. На взлом перебором уйдет миллионы лет.

**Q: Защищено ли соединение?**
A: Да, все данные передаются через HTTPS с TLS 1.3 — современный стандарт безопасности.

**Q: Есть ли лимиты на использование?**
A: Да, для защиты от атак: до 100 запросов в минуту, до 30 шифрований в минуту.
